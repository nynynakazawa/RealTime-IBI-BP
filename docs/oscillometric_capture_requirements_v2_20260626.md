# 押し込みオシロメトリ＋VFE 要件定義 v2（2026-06-26）

v1（`oscillometric_capture_requirements_20260624.md`）を土台に、今回の方針決定を反映して精緻化する。
**最終ゴール：追加センサーゼロ・スマホ単体・（ほぼ）どの機種でも、校正なしで絶対SBP/DBPを出す。**

## 0. v1からの主な変更点（先に差分）

| 項目 | v1 | **v2（本書）** |
| --- | --- | --- |
| 圧の代理 | 接触面積（主） | **VFE＝振動モーター＋IMUで力を推定**（主）。接触面積は補助に降格 |
| オシロメトリの目印 | 包絡振幅のピーク/消失 | **波形そのもの**：振幅ピーク=MAP、**波形の崩れ始め=SBP**（形態の破綻を一次目印に） |
| 絶対mmHg化 | 機種校正せず学習で復元 | **CNAPを答えに end-to-end 学習**（力の絶対値を明示推定しなくてよい）。dev時の力参照は任意の補助 |
| dev/deploy分離 | 曖昧 | **明確化**：dev時は力参照（はかり/ロードセル）OK。deploy時はスマホ単体・追加センサー無し |
| 確認済み事実 | — | **振動を与えても脈波(PPG)は崩れない**（実機で確認済）＝振動とPPG同時取得は成立 |
| 評価設計 | 相対→絶対の2段 | **3アーム・アブレーション**（受動のみ/能動のみ/融合）を追加 |

## 1. システム全体像（信号フロー）

```
スマホ単体で取得（追加センサーなし）
  ├ カメラPPG（緑/RGB平均）  → 脈波：①受動PP特徴 ②押下時の振幅包絡＋「崩れ」
  ├ 振動モーター（駆動）      → 一定振動を与える
  ├ IMU（加速度3軸＋ジャイロ3軸）→ 振動の鈍り＝VFE（力の情報）
  └ タッチ（接触面積/重心/長短径）→ 補助（品質ゲート・参考）
                    │
                    ▼ 学習（答え＝CNAP血圧、被験者単位LOSO）
        絶対 SBP / DBP（MAP+⅔PP, MAP−⅓PP で再構成 or 直接回帰）
```

- **役割分担**：
  - **目印（どこがMAP/SBP）＝カメラPPGの波形**（ピーク＝MAP、崩れ始め＝SBP）。
  - **力の情報＝IMU（VFE）**（明示的にN換算しなくても、特徴量としてCNAPに学習）。
  - **接触面積＝補助**（品質ゲート・正規化の参考。x軸主役からは外す）。

## 2. 制約・前提（不変条件）

- **C1 追加センサー禁止（deploy）**：外付け力センサー・はかり・カフを使わない。振動モーター/IMU/カメラ/タッチ＝**全機種内蔵のみ**。
- **C2 スマホ単体（deploy）**：測定時に外部機器なし。
- **C3 ユーザー校正なし**：被験者は事前にカフ等で自分の血圧を測らない。
- **C4 機種可搬性**：廉価版を除きほぼ全機種で動作。Pixel固有API（力センサ/温度計/画面下指紋）に依存しない。
- **C5 dev時の例外**：**開発・学習・検証の段階に限り**、力参照（はかり/ロードセル）やCNAPを使ってよい。これらは製品には入らない。
- **C6 後方互換**：既存CSV列順を壊さず末尾追加。既存解析PGが動く。
- **不変の参照**：**CNAP連続血圧**＝研究用の「答え」。学習・評価にのみ使用。

## 3. 取得データ schema（v1に IMU/vibration を追加）

### 3.1 per-frame（Wave_Data 相当, ~30fps）
v1の列（経過時間, Green, Red, Blue, DC_Green, contact_area, touch_pressure, touch_cx/cy, touch_major/minor, phase, press_target, touch_valid, SinWave）に**以下を追加**：

| 列 | 内容 | 取得元 |
| --- | --- | --- |
| vib_on | 振動ON/OFF（0/1）と強度（0–1） | アプリ状態 |
| imu_ax, imu_ay, imu_az | 加速度3軸（生 or 高周波成分） | IMU |
| imu_gx, imu_gy, imu_gz | ジャイロ3軸 | IMU |
| imu_ts | IMUサンプルのタイムスタンプ（同期用） | IMU |

※ **IMUはカメラ(30fps)より高レートで回る**（§4）。実装は「IMUを別バッファに高レート蓄積し、frameタイムスタンプで突き合わせて per-frame に代表統計（rolling SD等）を書く」か、「IMU生ストリームを別CSV `{session}_IMU.csv` に分離保存」のどちらか。**生IMUは別CSV分離を推奨**（レート差を壊さない）。

### 3.2 per-beat（Training_Data 相当）
v1の追加（contact_area/touch_pressure/RGB/DC_Green の代表値＋phase）に加え：

| 列 | 内容 |
| --- | --- |
| imu_rsd_a, imu_rsd_g | その拍区間のIMU加速度/ジャイロの rolling SD 代表値（VFE特徴の素） |
| ppg_ac_amp | その拍のPPG振幅（peak-to-trough、DC正規化後） |
| wave_break_score | **波形の崩れ度**（§5の形態破綻スコア） |
| press_phase_progress | 押下スイープ内の進捗（時間 or 接触面積基準） |

### 3.3 生IMU専用ファイル（推奨）
`{session}_IMU.csv`：`imu_ts, ax, ay, az, gx, gy, gz, vib_on`。高レート。frameと `imu_ts`/`経過時間_秒` で結合。

## 4. 振動・IMU（VFE）要件

- **R-VFE1 振動駆動**：押下相（phase2）で**一定の連続振動**を与える。Android `Vibrator`/`VibrationEffect`（amplitude指定, API26+）。強度は固定既定値（後で最適化）。
- **R-VFE2 IMUレート**：**できる限り高レート**（`SENSOR_DELAY_FASTEST`）。**目標 ≥400Hz**（振動基本波 ~150–235Hz の振幅変調を捉えるため）。
  - **必須：`HIGH_SAMPLING_RATE_SENSORS` 権限を Manifest に宣言**。Android 12+ は未宣言だと**200Hzに制限**される（通常権限・許可ダイアログ無し）。
  - **必須：端末のマイク・アクセスがON**。OFFだと権限を宣言しても動作センサが強制レート制限される（Android仕様）。
  - **受け入れ：計測開始時に“実効サンプリングHz”をタイムスタンプ差から実測し、ログ＋画面表示**（公称値でなく実際の達成レートを確認）。
  - 機種前提：**Pixel 8（無印）**。Bosch IMUでハード上限 ≥400Hz の見込み（実効値はアプリで確認）。
- **R-VFE3 特徴**：IMU各軸の **rolling standard deviation**（窓は拍 or 固定時間）を主特徴とする（VFE原理）。生も残す。
- **R-VFE4 PPGとの同時性**：振動ONのままカメラPPGを取得（実機で非干渉を確認済）。ただし**干渉の定量**（振動ON/OFFでのPPG SNR比較）をログから検証可能にしておく。
- **R-VFE5 同期**：カメラframe・IMU・タッチ・振動状態を**共通時計（経過時間_秒 or systemNanoTime）**で揃える。ズレ評価のため各ストリームに生タイムスタンプを残す。

## 5. 波形ランドマーク検出要件（オシロメトリの目印）

押下スイープ中のPPGから、以下を検出できるようにデータを残す（検出ロジックはPC側解析で実装可、端末はスコアの素を記録）。

- **L1 振幅包絡**：拍ごと `ppg_ac_amp`（DC正規化）。押下に対し**山なり（単峰）**を期待。**ピーク位置＝MAP点**。
- **L2 崩れ始め（破綻）＝SBP点**：動脈が潰れ始めると波形が**クリップ/平坦化/ノッチ消失/立上り鈍化**する。これを定量する **wave_break_score**（例：基準拍テンプレートとの相関低下、歪度/尖度変化、収縮期勾配の急減）。**破綻オンセット＝SBP/閉塞点**。
- **L3 消失点**：`ppg_ac_amp` が閾値以下＝完全閉塞（SBP上限の確認）。
- **L4 下側膝＝DBP点**（任意・弱い）：包絡の低圧側の比から。
- **x軸の与え方（優先順）**：①VFE力（IMU由来）→ ②押下時間（一定速度前提）→ ③接触面積（補助）。**absoluteはCNAP学習で復元するので、x軸は「相対的な押下進行」で十分**。

## 6. セッション・プロトコル（相＋寒冷昇圧＋3アーム対応）

1セッション＝以下の連続。`phase`列に記録。

- **phase0 配置**：品質ゲート（PPG拍動あり/飽和なし/位置OK）通過まで進まない。
- **phase1 安静PPG（受動用）**：数十秒。**受動PP特徴（sys_upslope/cosφ）の素**。
- **phase2 押し込みスイープ（能動用, 振動ON）**：ガイドで**一定速度**に押下（10–20秒）。複数回反復（包絡再現性）。
- **phase3 解放**：ゆっくり戻す（ヒステリシス確認）。
- **寒冷昇圧条件**：上記を**安静ベースライン**と**寒冷昇圧（昇圧後）**の両状態で行い、**CNAP血圧が変わった時にランドマーク（MAP/SBP点）が動くか**＝トラッキング検証の土俵を作る。**BPが動かないと追従は検証できない＝寒冷昇圧は必須**。

各相の時刻・品質ゲート結果を `{session}_meta.json` に記録。

## 7. dev時の力参照（VFE学習・検証用, 任意・製品外）

- **目的**：VFE（IMU→力）の妥当性確認と、必要なら明示的な力モデルの学習。
- **手段**：はかり/ロードセルを**dev時のみ**併用。スマホをはかりに乗せ、振動ONで段階押下→ **(IMU特徴, 真の力g)** を収集。
- **per-model プロファイリング**：機種ごとに5人程度×短時間でIMU↔力プロファイルを作成（Mukkamala VFEに準拠）。**開発側1回・ユーザー無関係**。
- **注意**：end-to-end（信号→CNAP血圧）学習なら**力の明示推定は必須ではない**。力参照は「VFEが効いているかのデバッグ／中間検証」と「機種差補正」のための**任意の補助**と位置づける。

## 8. 学習・評価要件（3アーム・アブレーション）

**同一の押しデータ（n=10〜20, CNAP参照）内**で対応比較。唯一の差＝特徴セット。

| アーム | 特徴 | データ源 |
| --- | --- | --- |
| **①受動のみ** | sys_upslope/cosφ 等 | phase1（安静）/連続のPPG |
| **②能動のみ** | 包絡ピーク・wave_break・IMU(VFE) | phase2（押下）|
| **③融合** | ①＋② | 両相 |

- **主張の主軸＝③ vs ①**（受動に押しを足すと SBP/DBP が**有意改善**）。**対応のあるLOSO・paired検定・効果量・CI**。n小は正直に。
- **②も入れて逆攻撃に備える**（③≥②＝受動PPの付加価値＝押しの合間を埋める）。
- **区間定義（交絡注意）**：アーム間差が「相の違い」と交絡しないよう、**同一参照CNAP値・整合した時間窓**に揃える。
- **目標2層**：Tier1＝**相対ΔMAP→ΔSBP/ΔDBPトラッキング**（主）、Tier2＝**絶対mmHg・AAMI/BHS**（副、崩れたら正直に「相対成立・絶対は今後」）。

## 9. UI要件（v1＋振動）

- v1のランディングゾーン／緑リング／信号ベース判定を踏襲。
- phase2で**振動ON**（ユーザーに振動が起きる旨を事前提示）。
- 一定速度押下ガイド（進捗バー）。痛み回避の**過圧上限で自動停止**。
- カメラ：**フロント＋画面緑照射**（v1）で確定。リア＋フラッシュ案は画面が見えず押下ガイド不可のため不採用。

## 10. 出力・自動エクスポート（v1＋IMU）

- セッション終了で `/sdcard/Download/` に自動保存：v1の全CSV＋**`{session}_IMU.csv`**＋`{session}_meta.json`。
- `Analysis/realtime_pipeline/auto_import_sessions.py` でadb自動pull＋merge（IMU CSVも対象に追加）。

## 11. 後方互換・非破壊（v1踏襲）

- 既存CSV列は末尾追加のみ。`merge_session.py`/既存解析が壊れないこと。IMUは別CSVで既存に影響させない。

## 12. 未決・要確認（正直に）

1. **IMU最大サンプリングレート**（機種依存）。VFEに足りるか（目標≥400Hz）。実機で計測必須。
2. **振動の基本周波数/強度**（LRA ~150–235Hz想定）。amplitude制御の安定性。
3. **振動×PPG干渉の定量**（非干渉は確認済だが、SNR比較で数値化）。
4. **wave_break_score の定義**（テンプレ相関低下 vs 勾配急減 等、どれが頑健か）。
5. **接触面積の飽和**：SBP閉塞前に面積が天井に達しないか。届かない時の押下ストロークの確保。
6. **指の柔らかさ×VFE感度**：押下レンジでIMU特徴が単調変化するか（はかりで dev 検証）。
7. **一定速度押下の再現性**：ガイドでどこまで揃うか（Tier1相対の前提）。
8. **n・寒冷昇圧での倫理/手順**：既存プロトコル範囲か確認。

## 13. 受け入れ基準（実装後）

- ビルド成功。phase2で**振動ON＋IMU高レート記録**＋PPG同時取得が実機で成立。
- `{session}_IMU.csv` が生成され、`imu_*` の値が押下で変化する。
- per-beat に `ppg_ac_amp`/`wave_break_score`/`imu_rsd_*` が出力される。
- 押下スイープでPPG振幅が**山なり→崩れ**を示し、ピーク/崩れ点が後処理で検出できる。
- 全CSV自動保存＋`auto_import_sessions.py` で取り込み＋merge。既存解析PGが動く。

## 14. de-risk 段階（更新）

1. **（済）振動×PPG非干渉**：実機で確認済。
2. **IMUレート確認**：≥目標Hzが出るか（§12-1）。
3. **VFE感度**：はかりでIMU特徴↔力が単調か（§12-6, dev）。
4. **オシロメトリ成立**：押下でPPGが山なり＋崩れ点が出るか（§5, GATE相当）。
5. **トラッキング**：寒冷昇圧でMAP/SBP点がCNAPと動くか（§6）。
6. **アブレーション**：③融合 > ①受動 が有意か（§8, 本丸）。
7. **絶対化（賭け）**：CNAP学習が between-subject で耐えるか（Tier2）。

各段で立たなくなれば主張を1段下げる（絶対→相対→A案フォールバック）。

## 15. 一行サマリ
**圧の代理は「接触面積」から「VFE（振動＋IMU）＋波形の崩れ」へ。力の絶対値は明示推定せずCNAPにend-to-end学習。追加センサーゼロ・スマホ単体・全機種を死守し、dev時のみ力参照を補助に使う。受動PP/能動MAP/融合の3アーム・アブレーションで「融合＞受動」を寒冷昇圧下・対応LOSOで示すのが評価の核心。**
